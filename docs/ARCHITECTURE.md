# MEMI — Architecture

> Regenerated 2026-07-05 from the actual code. Three apps, one repo, one MySQL database.

## System map

```
                         ┌───────────────────────────────┐
  shopper ──HTTPS──────▶ │ ecommerce (nginx static)      │──┐
                         │ "Memi Abbigliamento/"         │  │  /api/* proxied
  admin  ──HTTPS──────▶  │ admin (nginx static) "MEMI/"  │──┤──────────────▶ backend (Express :3000) ──▶ MySQL 8
                         └───────────────────────────────┘  │                     │
  Stripe webhook ────────────────────────────────────────────┘                    ├─ uploads volume (product images)
                                                                                  └─ SMTP (nodemailer) / Stripe API
```

- **Storefront** (`Memi Abbigliamento/`): static HTML/CSS/JS served by nginx. Catalog pages hydrate at
  runtime from `GET /api/products` via `catalog-loader.js`; product detail is the single dynamic
  `/product?id=<slug>` page (old `products/<slug>/` pages are meta-refresh redirects to it).
  Cart/wishlist in localStorage (`memi_cart`, `memi_wishlist`), synced to the API for logged-in users
  (debounced PUT `/auth/cart` / `/auth/wishlist`). Checkout = Stripe Elements + gift cards + discounts.
- **Admin** (`MEMI/`): jQuery SPA, single `dashboard.html`, hash-routed views. `js/admin-api.js` wraps the
  API (token `memi_admin_token`); `js/app.js` renders views via the `_origRenderView` override pattern
  (fetch → populate `DATA` → call original renderer). Role-gating: `ADMIN_ONLY_VIEWS` + staff nav hiding.
- **Backend** (`MEMI-Backend/`): Express + mysql2/promise pool. Routes in `src/routes/` (see docs/api.md),
  auth middleware in `src/middleware/auth.js`, zod validation (`src/validation.js`) on highest-risk routes,
  Pino logging, Helmet, CORS whitelist, 3-tier rate limiting, sharp image pipeline (`src/images.js`),
  nodemailer (`src/email.js`, silent no-op without SMTP_USER), audit log (`src/audit.js`).

## Data flow: product from admin to shop
1. Admin creates product (`POST /api/products`) + uploads images (`POST /api/products/:id/images` → sharp → webp trio in uploads volume, served at `/api/uploads/<hash>-{card,full,thumb}.webp` with immutable caching).
2. Shop pages fetch `GET /api/products?...` at runtime → cards link `/product?id=<slug>` → `product.html` hydrates detail + reviews.
3. No rebuild needed for products. (Collection page *hero/filter-count chrome* is still build-time generated — see Known drift.)

## Database
MySQL 8, utf8mb4. Core tables from `schema.sql` (admin_users, customers, customer_addresses, products,
product_sizes, orders, order_items, couriers, shipments, shipping_zones, discount_codes, discount_usage,
newsletter_subscribers, invoices, resi, reviews, store_settings) + feature tables ensured on every boot by
`db/migrations.js` (gift_cards, campaigns, cms_pages, blog_posts, pickup_points, loyalty_transactions,
audit_log) + idempotent column/index additions. Schema self-heals structurally on boot; seed data only on
fresh volume or `npm run db:init`.

## Payments integrity
- Client gets intent from `/payments/create-intent`; server verifies on order create: intent succeeded,
  amount == server-computed total (exact cents), currency EUR; `orders.payment_intent_id` UNIQUE (no replay).
- Zero-total orders (full gift card) skip Stripe, auto-`pagato`.
- Webhook (raw-body, signature-verified) logs orphan payments and disputes.

## Caching & asset versioning (fully automated)
- Both nginx configs: HTML → `Cache-Control: no-cache, must-revalidate`; JS/CSS/img → 30d immutable.
- Both Dockerfiles run a build stage `node scripts/cache-bust.js .` that rewrites every local `.js`/`.css`
  `?v=` reference in HTML to an 8-char content hash (auto-discovery, relative-path aware, idempotent).
  Manual `?v=N` bumps are no longer required for deploys; source values only need to stay consistent
  (checked by `verify/run.sh`).

## Known drift / legacy
- `productsData.js` is dead at runtime (kept for reference).
- `collections/<slug>/index.html` chrome (hero text, filter counts) is generated by
  `scripts/generate-collections.js` reading the API (`MEMI_API_BASE`); product grids self-heal at runtime,
  counts drift until regeneration.
- Some header/footer links use `/shop?categoria=…` while others use `/collections/<slug>/`.

## Environments
- Local: `docker compose -f docker-compose.yml -f docker-compose.local.yml up --build`
  (shop :8080, admin :8081, api :3000, mysql :3307). Zero secrets needed (dev JWT defaults in compose;
  Stripe 503s; emails no-op).
- Production: Hetzner + Coolify + Traefik (labels in docker-compose.yml), domains via `SHOP_DOMAIN`,
  `ADMIN_DOMAIN`, `API_DOMAIN`. See docs/DEPLOYMENT.md.

## Verification
- `bash verify/run.sh` — no-DB gate: JS syntax (all files), `?v=` consistency, route contracts
  (verify/contract.cjs), order-flow/webhook/gift-card simulations, zod schema tests.
- `./smoke-test.sh` — live-stack gate. `./run-live.sh` — hits a running stack. `e2e/` — Playwright.
- Backend unit tests in `MEMI-Backend/test/` (node --test).
